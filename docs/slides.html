A Project Manager
Simulation Environment

One simulated week at a SaaS company. An agent is the PM; simulated coworkers do the work.

rule #1 LLMs only produce dialogue — every consequence comes from deterministic rules result same seed + same actions = the same week, byte for byte (stable rewards for GRPO; every run replayable)

1 · the world

Time jumps between events — nothing else exists

Mon 09:05 you ping Sarah ← your action costs 1 sim-minute Mon 09:06 you skip ahead 3 hours… Mon 09:42 …interrupted: Sarah replies ← her latency: a seeded 5–45 min draw Mon 09:42 the model thinks for 30 real seconds — sim clock doesn't move Tue 09:00 overnight, the migration gained 0 hours: nobody works at night — progress is computed from the calendar, no process ever "runs"

Every change is one line in a log — any past instant can be rebuilt, no LLM needed. Nothing is polled: chats wake you, email arrives on its own schedule.

1 · the world — communication

Three channels, three price tags

chat 10:00 ping Sarah 10:01 ping again 10:02 ping again → she answers in 5–45 min → but loses 3 × 20 min of deep-work focus

interruptions stack — they don't overlap

email 10:00 email Sarah 11:17 she reads it at her next inbox pass → slower answer → zero focus cost

async by social contract; the agent's own inbox check fires every 30 min

meeting 14:00 1h sync, 4 people → instant answers, minutes kept → costs the org 4 person-hours

the most expensive artifact in any company

These numbers appear in no tool description. Agents learn them three ways: timestamps, complaints from over-pinged coworkers, or lower scores.

1 · the world — why agents can't just pick one channel

Each channel has a scenario it strictly wins

chat wins Fri 09:30 Sarah's task is STALLED until you answer one question. chat → reply lands at once → unblocked. 20-min tax ≪ the hours she'd sit idle email → waits ~2h for her inbox → the morning is gone meet → a room booked to say one sentence
email wins Thu 09:00 execs want sign-off notes from BOTH engineers by end of day. No rush. email → zero focus cost, answered at next pass — hours of slack to spare chat×2 → 40+ min of focus stolen mid-crunch → completions slip meet → 3 person-hours for something async
meeting wins Wed the migration IS the critical path; Sarah alone misses Friday. 2h session on the task: 1.3 + 0.6×1.0 = 1.9× → 3.8h banked → ships Thu chat/email → words move, work doesn't — no channel adds hands but a session

No channel always wins → no one-channel shortcut survives. The loss comes from physics, not a judge. Measured: an always-chatting PM scores below zero.

1 · the world — tasks

New work lands on someone — usually the wrong someone

Tue 11:30 incident arrives → auto-assigned to Dave (the yes-to-everything volunteer) Tue 11:30 Dave pings you: "SSO is down for Meridian — I've got it 👍" (ticket: sso-4f2a, currently on dave) Dave's queue: 30h. Sarah frees up Wednesday. your move A: Tue 11:32 reassign → Sarah → ships Thu 15:00 your move B: Thu 09:00 reassign → Sarah → work only counts from Thursday → misses Friday your move C: do nothing → Dave gets to it eventually → the unmanaged week

People self-schedule by priority; half-done work is never lost. Your levers: move a task or reorder a queue (relabeling can't inflate the score). Option C — the unmanaged week — is the score's zero point.

2 · belief updates

Every fact exists three times

the migration, Monday 09:05: truth 55% done ← only the engine & the grader can see this belief Sarah: "70%, on track" ← honest. wrong. she hasn't re-checked. the board "70%, on track" ← inherited her stale claim ask her Monday → you get an honest 70%. there is no oracle to interrogate. and Dave insists: "rate-limiting is launch-critical!" ← authored red herring; it isn't. chasing it burns real capacity.

Belief changes are scheduled events — no LLM touches information. The board self-corrects only on completions, the one reliable public signal.

2 · belief updates — one task's week

What the agent can know, and when

Mon 09:05 Sarah's estimate: "~8h left" honest. wrong. truth: ~36h. Tue 14:00 belief-update event fires → Sarah realizes it's ~36h she pings you, unprompted: "the SDK integration blew up — ~36h left, not 8" Wed 10:00 you ask again → her honest current estimate (still not truth) Thu 16:30 migration actually completes → board flips to Done two completions ≈ a speed estimate for that person

Estimates are in hours-left, not %, and the blind spot belongs to the person: reassign the task to a fresh owner and Sarah's stale number doesn't follow. The graded skill — keep slack, react fast to confessions, calibrate people from what actually ships.

3 · skills

Some people are quietly faster — nobody tells you

config (frozen numbers, never a runtime LLM): sarah.skills = {db: 1.3} dave.skills = {auth: 1.3} the same 12h database task: on Dave → 12h of calendar on Sarah → 9¼h (12 ÷ 1.3)
what the agent sees: board: nothing about skills tools: nothing about skills what the agent can notice: Mon gave Sarah the db task Thu she finished a day "early" → infer she's the db specialist → route the next one accordingly

Same physics for every rollout — stable rewards, nothing an agent can talk its way into. The perfect week routes by skill, so discovering specialists is part of the job.

3 · skills — more than one person on a task

More hands help — but only a meeting makes them count

migration is the bottleneck (Sarah's, db). three ways to add Dave: solo Sarah → 1.3× +Dave, no meeting → 1.5× both hack at one task, uncoordinated — they conflict & duplicate. barely helps. +Dave, 2h session on the task → 1.9× 1.3 + 0.6×1.0, banked at meeting-end. the sync is what makes his hands land.

solo < parallel < meeting — the meeting buys that last jump.

the fine print: hands cap out — 1, 0.6, 0.36, … parallel ≤ 1.5×, meeting ≤ 2.5× never linear in headcount. the bill (either way): Dave's own 30h queue is frozen while he's on Sarah's task — a meeting also books a room-hour for everyone in it. → worth it for a real bottleneck or an idle teammate; a net loss otherwise.

“Adding manpower to a late software project makes it later.” — Fred Brooks, The Mythical Man-Month (1975). The two caps and the fading weights are that law as physics.

4 · the rubrics

Every task instance carries its own scale

< 0 made the week worse
pings, churn, busywork
0 = the unmanaged week
defaults play out, agent does nothing
1 = best solo week
one owner per task, zero friction — a reference, not a ceiling
both anchors computed from the config file — before any agent runs: unmanaged week ships 58% of the work-weight → score 0 best solo week ships 100%, early → score 1 (one owner/task) an agent ships 79%, mostly early → score 0.41 score > 1 is real, not a bug — the agent out-coordinated the solo optimum: pulling idle teammates onto the bottleneck (parallel work, or a session) — a lever a one-owner schedule can't use. bounded by the caps. never clamped.

Why an unbeatable ceiling isn't needed: the score is an affine map of COMBINED, and GRPO advantages are affine-invariant (next slide) — so 1 only has to be a cheap, stable anchor. The gap's width IS the difficulty: how wrong the week's defaults are.

Every lever is a number in a config file. One bound on all of them: a task must stay completable from the moment the PM could know of it — otherwise the instance is rejected.

4 · the rubrics — what is measured

Outcomes only — and it already separates models

COMPLETION = Σ w·(progress + α·done)/(1+α) the work got done EFFICIENCY = Σ w·done·(horizon − done_at)/span …and landed early COMBINED = (COMPLETION + γ·EFFICIENCY)/(1+γ) the reward scalar FAIRNESS = 1 − σ(hours worked / available) team health, per person SCORE = (agent − baseline)/(OPT − baseline) position between the anchors = a·COMBINED + b, a > 0 fixed per task an affine transform
termmeaning
wauthored priority weight — P0 8 · P1 4 · P2 2 · P3 1 (agent-created tasks: w = 0)
progress, donetrue fraction of effort completed ∈ [0,1]; done ∈ {0,1}
α = 0.5finishing bonus — a shipped task beats an almost-done one
γ = 0.5earliness weight in the combined scalar
horizon, spanFri 17:00; span = horizon − Monday start
σspread (std dev) of workload across the workers

GRPO's group normalization (r − mean)/std is invariant to affine transforms — within one task, SCORE and raw COMBINED yield identical advantages. The normalization changes nothing for training; its job is to put hundreds of tasks on one scale.
