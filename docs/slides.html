project presentation

A Project Manager
Simulation Environment

One simulated work week at a small SaaS company. An agent plays the project manager; simulated coworkers do the work. The design rule throughout: LLMs only produce dialogue — every consequence in the world comes from deterministic rules. That makes runs replayable, rewards stable across rollouts (GRPO-clean), and the score impossible to talk your way into.

1 · the world

A week that runs on events, not ticks

  • Time is a single clock that jumps from event to event (a discrete-event simulation). Each agent action costs one simulated minute; model inference latency never touches simulated time.
  • Task progress is computed, not simulated: a pure function of the config, the clock, and the action log — work "accrues overnight" with no process running.
  • Every state change is one line in an append-only log, so any past instant can be rebuilt exactly, with zero LLM calls.
  • Nothing is polled — everything is delivered. The agent has no background thread, so "checking your inbox" is itself a scheduled event: a chat ping wakes the agent even mid time-skip; email arrives when the next inbox-check event fires.
  • Determinism contract: same config + same seed + same actions = the same week, byte for byte. All randomness is drawn from seeded streams and frozen into the scenario file.
1 · the world — communication

Channels are priced like a real office

channelspeedcostthe real-life claim behind the number
chat answered in 5–45 sim-minutes; reaches the agent instantly each message costs the receiver 20 minutes of focus — and pings stack, they don't overlap an interruption costs ~20 min of refocus on deep work; three pings are three broken flows
email read at the receiver's next inbox pass (~every 1.5–2.5 h; the agent's fires every 30 min) zero focus cost email is async by social contract — nobody drops their editor because mail arrived
meeting instant back-and-forth in a live room; verbatim minutes kept blocks every attendee's calendar for the whole slot a 1-hour meeting with 4 people costs the org 4 person-hours

Crucially, none of these numbers appear in any tool description. Agents learn the economics from evidence (reply timestamps, completions slipping), from people (the focus cost is felt, so coworkers complain in character when over-pinged), or from reward — noisy trajectories just score lower.

1 · the world — tasks

New work lands on someone — usually the wrong someone

  • Mid-week, external tasks arrive (an incident, a legal request). Each one lands already assigned to a default person — often the office volunteer who says yes to everything — and the agent is told: "ticket X, currently on Dave".
  • The manager's two levers: move it (reassign to a better owner) and reorder it (raise or lower where it sits in someone's queue — which changes scheduling only, never scoring weight, so relabeling can't inflate the score).
  • Coworkers self-schedule sensibly: everyone always works their highest-priority available task and switches when that changes; paused work resumes exactly where it stopped.
  • Acting late costs: a task moved on Thursday only accrues work from Thursday — there is no retroactive credit for decisions you didn't make on Monday.
  • If the agent does nothing all week, the defaults just play out — that whole "unmanaged week" is the zero point of the score.
2 · belief updates

Three versions of every fact

layerwho holds itbehavior
ground truththe engine and the grader — nobody in the sim the scheduler's actual state; not even the person doing the task can read it
beliefeach coworker, about their own work honest but often wrong — and it updates on an authored schedule: Sarah genuinely believes "70%, on track" on Monday and realizes "55%, schema rework" on Tuesday, sometimes messaging the manager to confess unprompted
the trackereveryone the shared board — lags behind, inherits stale claims; it self-corrects only on completions, the one public, reliable signal in the whole environment

So "just ask everyone on Monday" collects honest wrong answers — there is no oracle to interrogate. Scenarios also plant red herrings: confidently-held false beliefs ("rate-limiting is launch-critical!") that cost real capacity if the agent chases them. Every update is a scheduled, deterministic event — no LLM sits on any information path.

2 · belief updates — one task's week

What the agent can know, and when

Mon 09:05 board says "~70%, on track" Sarah believes it truth: 55% Tue 14:00 scheduled belief-update fires → Sarah realizes 55% she pings you: "schema rework surfaced — ~18 focused hours left" Wed–Thu asking again gets her honest current belief (still not truth) Thu 16:30 the task actually completes → board flips to Done ← the reliable signal: two completions ≈ a speed estimate per person

The skill being graded is not truth-extraction — it's operating under honest noise: keep slack against estimates, react fast to confessions, calibrate people from what actually ships.

3 · skills

Some people are quietly faster — nobody tells you

config (frozen at authoring): sarah.skills = {db: 1.3} dave.skills = {auth: 1.3, ops: 1.2} task.tags = ["db"] speed(person on task) = geometric mean of matching factors → Sarah does db work 30% faster
  • Skill factors are plain numbers in the scenario file — never a runtime LLM call. Every rollout of a task sees identical physics (clean group-relative advantages), and there is nothing an agent can phrase its way into.
  • Hidden from the agent: not on the board, not in any tool output. The only way to learn who's fast at what is to watch how quickly work actually completes.
  • The scoring ceiling routes by skill too, so "give the db task to the db person" is part of what a perfect week means.
3 · skills — combined in working sessions

Meetings can move work — boundedly

a task has exactly ONE owner. more hands = book a meeting ON the task: schedule_meeting(..., task="migration") session rate = best contributor + 0.6 × next + 0.36 × next … capped at 2.5× the session's output lands on the task when the meeting ends
  • Attendees' skill factors combine with diminishing returns: a war-room can roughly double the pace of a bottleneck task — it can never scale linearly with headcount (unbounded parallelism would break the scoring ceiling; this is Brooks's law as physics).
  • The trade is genuine: the session's boost vs everyone in the room stopping their own work for the slot.
  • So the winning move is narrow and real: swarm the skill-matched people on the critical-path task — and nothing else. Idle syncs are pure loss, and the score says so.

“Adding manpower to a late software project makes it later.” — Fred Brooks, The Mythical Man-Month (1975). The cap and the diminishing weights are that law written as physics: people help, but coordination overhead eats each additional pair of hands.

4 · the rubrics

Every task instance carries its own scale

< 0 made the week worse
interruptions, churn, busywork
0 = the unmanaged week
same sim, agent does nothing
1 = the perfect week
frictionless optimum, exhaustive search
  • Both anchors are computed from the scenario file alone — before any agent runs — and stamped into it. Score = the fraction of the gap between them the agent actually captured.
  • That normalization is what makes hundreds of generated task instances comparable — and the gap's width is the authored difficulty: it measures how wrong the week's default assignments are. A gate rejects any instance where an unmanaged week is already near-optimal.
  • Instances are generated from a template by remixing physics (efforts, timings, default owners, ticket ids) under seeded draws, then gated for validity, fairness, and difficulty tier.
4 · the rubrics — what is measured

Outcomes only — and it already separates models

metricplain reading
completiondid the work get done (bigger/higher-priority tasks count more)
efficiencyhow early finished work landed before the Friday deadline
done-rateweighted fraction of the week's work that shipped
fairnesswas load spread across people, or piled on one person

No keyword checks, no LLM judge, no activity credit — the only path to a score is a genuinely better week. Interruptions and idle meetings lower it through capacity physics alone.

the pipeline

From one authored company to a graded curriculum

template ─→ generate (seeded physics remix) ─→ gate (valid · fair · difficulty tier) ─→ stamp the two anchors ─→ run models ─→ live dashboard

  • Difficulty is measured, never asserted: the config-side fingerprint predicts it; model score-degradation across tiers confirms it.
  • Deterministic end to end — suited to training, not just evaluation: fixed physics per instance, always-numeric reward, reproducible weeks.
  • Every benchmark point is three clicks from its evidence: task → run → the agent's full conversation (every thought, tool call, and result, untruncated) next to the world it produced.
